# ESC — 설계 개요 (제품 + 방법론)

> ESC = 학생용 인지·심리 훈련 앱. EndeavorRx식 적응형 게임 원리를 빌리되,
> **의료 주장 없이** "집중력·학습 향상" 비의료 제품으로 출시하고, 추후 식약처
> 디지털치료기기(DTx) 전환 옵션을 위한 데이터·로깅 설계는 처음부터 깐다.
>
> 구조 결정(2026-06-10): **한 통합게임이 아니라 종목(기둥)별 따로 모드/미션.**
> "오늘은 듣기 모드, 내일은 집중 모드"식. 6개 모드 + 이를 묶는 공유 셸.

---

## 1. 6개 훈련 종목(모드)

| # | 모드 | 층 | 훈련 기둥 | 핵심 기전 | 상태 |
|---|------|----|----------|----------|------|
| 1 | **듣기 모드** | 인지 | ①듣기능력 (+③속도, ⑤끈기) | 적응형 청해(기존 오디오150 재활용) | 자산·알고리즘 보유 → **첫 빌드** |
| 2 | **집중 모드** | 인지 | ②집중력 | 듀얼태스크(조종+변별 동시), 간섭비용 감소 | 신규(EndeavorRx 코어) |
| 3 | **속도 모드** | 인지 | ③속도 | 반응시간 압박·빠른 판단 | 신규(경량) |
| 4 | **도전 모드** | 정서 | ④두려움(문제풀때) | 안전한 모의압박 점진 노출, 무해한 실패 | 정서 레이어→독립모드 |
| 5 | **끈기 모드** | 정서 | ⑤포기 | 스트릭·"한 번 더"·컴백 보너스 | 정서 레이어→독립모드 |
| 6 | **진정 모드** | 정서 | ⑥심리안정 | 호흡·그라운딩 루틴 (모드 사이 인터루드로도 재사용) | 신규(경량) |

정서 3종(④⑤⑥)은 초기엔 인지 모드 위의 **경험설계 레이어**로 구현하고, 필요 시
독립 모드로 승격한다(과잉구축 방지). 진정 모드만 처음부터 독립 + 공용 인터루드.

## 2. 공유 셸 — "6앱이 아니라 1앱"으로 만드는 것

- **홈/모드 메뉴**: 6모드 카드 + "오늘의 추천 모드"(요일 로테이션 또는 약점 기반). 미구현 모드는 "준비중".
- **공통 적응엔진**: ~80% 정확도 목표 난이도 컨트롤러. 모드별로 파라미터만 주입.
- **공통 진행·보상**: 스트릭·레벨·뱃지(초기 localStorage).
- **공통 텔레메트리**: 가명(익명ID) 이벤트 로그. 초기엔 console/localStorage, 추후 Firehose→S3 자리만 비워둠.
- **공통 진정 인터루드**: 세션 끝/모드 사이 호흡 루틴(⑥).

### 모드 모듈 계약 (셸 ↔ 모드 인터페이스)
모든 모드는 다음을 셸에 노출한다(언어 무관, JS 초기 / Flutter 포팅 시 동일 형태):
```
mode = {
  id, title, pillars[],          // 메타
  start(config, ctx) -> session, // ctx = { log(event), adapt, onComplete }
  // session 진행 중 ctx.log(...)로 텔레메트리 발화
  // 종료 시 ctx.onComplete(result) 호출 → 셸이 보상·인터루드·홈복귀 처리
}
```

## 3. 작업 방법론 — 종목별 Phase 1~4 (각 Phase ≥7단계, 끊임없이 검증·수정)

모든 모드는 동일한 4-Phase 템플릿을 따른다.

- **Phase 1 — 기획·설계**: 치료목표·기둥매핑 / 자산·코드 인벤토리 / 적응 파라미터 확정 / 셸 계약 / 텔레메트리 스키마 / UI 플로우 / 성공기준·검증항목 / 비의료 카피 가드.
- **Phase 2 — 코어 구현**: 리팩터 구조 / 화면·라우팅 / 코어 루프 / 적응엔진 연결 / 인터루드 / 진행·보상 최소 / 텔레메트리 로거 / 로컬 동작 확인.
- **Phase 3 — 적응·정교화·검증빌드**: 시뮬레이터 / 적응 튜닝 / 견고화(로드·엣지) / 반응형 / UX 마감 / 비의료 감사 / 텔레메트리 발화 검증.
- **Phase 4 — 검증·튜닝·출하**: E2E(Playwright) / 수렴 리포트 / 플레이테스트 / 버그 루프 / 성능 / 배포 / 라이브 검증 / 회고+다음 모드 착수.

**원칙**: 각 단계마다 verify 항목을 정하고, 통과 못 하면 다음으로 넘어가지 않는다.

## 4. 빌드 순서 & 플랫폼

1. **듣기 모드부터** — 오디오·알고리즘 자산이 이미 있어 "셸 + 첫 모드"를 최저비용으로 띄움. 이걸로 공유 셸을 확립.
2. **집중 모드** — 셸에 꽂는 첫 신규 게임. 여기서 "재미 + 80% 수렴"을 검증.
3. 이후 속도 → 진정 → 도전 → 끈기.

**플랫폼**: Phase 0 = 웹 스파이크(현 Desktop\ESC 정적 웹 그대로 확장)로 코어 루프·재미·튜닝을
싸게 검증 → 검증되면 **Flutter + Flame**로 정규화 포팅. (관련: 신규 개발자의 Flutter/AWS 방향과 일관.)

## 5. 비의료 가드레일 (전 모드 공통, 위반 금지)

- 진단·치료·"ADHD 검사/테스트"·"TOVA"·임상 점수 해석을 사용자에게 노출 금지.
- d′·criterion·CPT 등 임상 지표는 **내부 텔레메트리 전용**(추후 DTx 전환용). 사용자에겐 "집중도/정확도/연속" 같은 비의료 표현만.
- 텔레메트리는 **가명**(PII 없음, 익명ID).
- 카피는 "학습·집중 훈련", "연습", "도전" 톤. "치료/완치/증상 개선" 금지.
